# Сборка, публикация и запуск проекта 

## Сборка проекта

Для сборки проекта выполните следующую команду из каталога с решением:

`$ dotnet build ./EventManagementService/EventManagementService.csproj`

## Публикация проекта

Для Windows в каталоге с решением выполнить:

`$ dotnet publish ./EventManagementService/EventManagementService.csproj -c Release -r win-x64`

Для Linux в каталоге с проектом выполнить:

`$ dotnet publish ./EventManagementService/EventManagementService.csproj -c Release -r linux-x64`

## Сборка и запуск проекта

Для сборки с последующим запуском проекта выполните следующую команду из каталога с решением:

`$ dotnet run --project ./EventManagementService/EventManagementService.csproj`

## Запуск тестов

Для запуска тестирования следующую команду из каталога с решением:

`$ dotnet test`

## Загрузка и установка зависимостей

Для загрузки и установки зависимостей проекта выполните следующую команду из каталога с решением:

`$ dotnet restore ./EventManagementService/EventManagementService.csproj`

# Работа со Swagger

Для работы со Swagger запустите проект и перейдите в браузере по следующему адресу:
`http://localhost:port/swagger/index.html`

На место `port` впишите порт, с которым запустился EventManagementService.

# Логика фоновой обработки

Для бронирования события используется эндпоинт `/events/{id}/book`, при работе с которым передается `id` события для бронирования. Если событие с указанным `id` есть, то клиенту возвращается ответ `202 Accepted`. В фоновом режиме выполняется задача обновления статусов бронирований в рамках Background-сервиса `BookingHostedService`. В этом сервисе, один раз в секунду, из базы данных, в которой хранится информация о бронировании, извлекаются все бронирования со статусом `Pending`, для каждого экземпляра создается задача имитирующая работу с внешней системой с паузой на две секунды. Выполняется ожидание всех запущенных задач, после этого обновляется статус всех бронирований на `Confirmed` с установкой значения для `ProcessedAt`.

# Использованные примитивы синхронизации

Для синхронизации доступа к коллекциям, с которыми работают `EventService` и `BookingRepository` используется блокировка `lock`.

В `BookingService` в методе `CreateBookingAsync` используется `SemaphoreSlim`, а не `lock`, т.к. в нем используются асинхронные методы сервиса `EventService` и репозитория `BookingRepository`.

В `BookingHostedService` в методе обработки броней `ProcessBookingAsync` также используется `SemaphoreSlim`, т.к. в нем используются асинхронные методы сервиса `EventService` и `BookingService`.

# HTTP API

## Получение списка всех событий

Хендлер: `GET /events`.

Список параметров запроса:
* title (string, опциональный) - поиск по названию (регистронезависимый, частичное совпадение)
* from (DateTime, опциональный) - события, которые начинаются не раньше указанной даты
* to (DateTime, опциональный) - события, которые заканчиваются не позже указанной даты
* page (int, опциональный, c значением по умолчанию 1) - страница, которую необходимо вернуть
* pageSize (int, опциональный, со значением по умолчанию 10) — количество элементов на странице

Возвращает `PaginatedResultResponseDto` со следующим набором полей:
* TotalEventsCount - общее количество событий
* Events - массив событий
* PageNumber - номер текущей страницы
* EventsCountOnPage - количество элементов на текущей странице

Возможные коды ответа:
* 200 - успешное получение списка событий.
* 400 - ошибка валидации входных данных. Ответ возвращается завернутый в `ProblemDetails`.

## Получение события по id

Хендлер: `GET /events/{id}`.

Возвращает событие по указанному идентификатору `id`.

Возможные коды ответа:
* 200 - событие найдено.
* 404 - событие не найдено. Ответ возвращается завернутый в `ProblemDetails`.

## Создание события

Хендлер: `POST /events`.

Создаёт новое событие.

Возвращает `EventResponseDto` со следующим набором полей:
* Title - название события
* Id - идентификатор события
* Description - описание события
* TotalSeats - общее количество мест на событии
* AvailableSeats - количество свободных мест на событии
* StartAt - дата/время начала события
* EndAt - дата/время завершения события

Возможные коды ответа:
* 201 - событие успешно создано.
* 400 - ошибка валидации входных данных. Ответ возвращается завернутый в `ProblemDetails`.

## Обновление события

Хендлер: `PUT /events/{id}`.

Обновляет существующее событие с идентификатором `id`.

Возможные коды ответа:
* 204 - событие успешно обновлено.
* 400 - ошибка валидации. Ответ возвращается завернутый в `ProblemDetails`.
* 404 - событие не найдено. Ответ возвращается завернутый в `ProblemDetails`.

## Удаление события

Хендлер: `DELETE /events/{id}`.

Удаляет событие по идентификатору `id`.

Возможные коды ответа:
* 204 - событие успешно удалено.
* 404 - событие не найдено. Ответ возвращается завернутый в `ProblemDetails`.

## Создание бронирования

Хендлер: `POST /events/{id}/book`.

Создает бронирование для события с идентификатором `id`. В заголовке `Location` возвращает ссылку на ресурс брони.

Возвращает `BookingResponseDto` со следующим набором полей:
* Id - идентификатор брони
* EventId - идентификатор события, для которого создается бронь
* Status - статус брони, в рамках данного запроса значение - 0. Возможные значения:
    ** 0: Pending - бронь создана, ожидает обработки
    ** 1: Confirmed - бронь подтверждена
    ** 2: Rejected - бронь отклонена
* CreatedAt - дата/время создания запроса на бронирование события
* ProcessedAt - дата/время, когда выполнена обработка запроса. В рамках данного запроса значение - пустое поле.

Возможные коды ответа:
* 202 - запрос бронирования принят на обработку.
* 404 - событие с указанным `id` не найдено. Ответ возвращается завернутый в `ProblemDetails`.
* 409 - нет больше свободных мест на событие с указанным `id`. Ответ возвращается завернутый в `ProblemDetails`.

## Получение текущего состояния брони по её идентификатору

Хендлер: `GET /bookings/{id}`.

Возвращает текущее состояние брони с идентификатором `id`. 

Возвращает `BookingResponseDto` со следующим набором полей:
* Id - идентификатор брони
* EventId - идентификатор события, для которого создается бронь
* Status - статус брони:
    ** 0: Pending - бронь создана, ожидает обработки
    ** 1: Confirmed - бронь подтверждена
    ** 2: Rejected - бронь отклонена
* CreatedAt - дата/время создания запроса на бронирование события
* ProcessedAt - дата/время, когда выполнена обработка запроса

Возможные коды ответа:
* 200 - информация о бронировании с указанным идентификатором есть и возвращена клиенту.
* 404 - бронирование с указанным `id` не найдено. Ответ возвращается завернутый в `ProblemDetails`.

# Сценарии использования

Пример сценария использования:
1. Пользователь получает список событий через `GET /events`, находит среди них подходящее и запоминает его `id`.
2. Выполняет бронирование события через `POST /events/{id}/book`, где `id` - это идентификатор интересующего события. Из заголовка Location либо тела ответа извлекает и запоминает id бронирования.
3. Периодически проверяет статус бронирования через `GET /bookings/{id}`, где `id` - это идентификатор бронирования.

Сценарий с овербукингом: если общее количество мест у события пять, а количество попыток бронирования (`POST /events/{id}/book`) больше пяти, то все последующие попытки забронировать место на такое событие будут сопровождаться выдачей кода ответа 409.
